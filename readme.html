End-to-end pipeline — bird's eye view
Five stages that transform two PDF files into a visual, word-level diff
Stage 1
File selection
User picks Old PDF + New PDF via file dialog. Both paths stored. Compare button enabled.
Stage 2
Extraction
pdfplumber native text → if < 20 words: pypdfium2 render + Tesseract OCR per page.
Stage 3
Chunk + Seam
10-page batches. Seam detection finds safe cut point (6+ consecutive identical words).
Stage 4
Two-pass Diff
Python difflib LCS first pass → C++ Kv spatial engine second pass.
Stage 5
Render + Display
pypdfium2 renders pages → PIL draws highlight overlays → tkinter side-by-side canvas.

Global configuration constants
Every tuning knob that drives behaviour across the whole system
RENDER_SCALE = 1.5PDF pages rendered at 1.5× native resolution. Higher = sharper highlights, more RAM. Used for both rendering and OCR image generation.
CHUNK_PAGES = 10Process 10 pages at a time. Prevents loading entire book into RAM. UI updates progressively as each chunk finishes.
SEAM_MIN_RUN = 6Require 6+ consecutive identical words to declare a safe chunk boundary. Prevents splitting mid-sentence.
SEAM_SCAN_PCT = 0.35Only scan the last 35% of each chunk for a seam. Balances thoroughness vs. performance.
OCR_MIN_WORD_COUNT = 20If pdfplumber extracts fewer than 20 words from a page → treat as scanned image and trigger OCR fallback.
OCR_MIN_CONF = 30Tesseract confidence threshold. Words scoring below 30/100 are discarded as likely misreads from poor scan quality.
PAGE_GAP = 2828px vertical gap between rendered page images on the canvas. Visual breathing room between pages.
HIGHLIGHT COLORS Yellow (255,213,79,115) = removed words in old PDF
Blue (79,195,247,115) = added words in new PDF
Alpha 115/255 = semi-transparent so text shows through.

3
Text extraction — extract_page_range()
Per-page routing: native pdfplumber or OCR fallback. Both produce identical output.
Per page: how many native words did pdfplumber find?
≥ 20 words → Native path
Use pdfplumber word list directly. Each word comes with text, x0, top, x1, bottom, page. Fast, pixel-perfect coordinates embedded in the PDF. No image rendering needed.
< 20 words → OCR fallback
Render page to image via pypdfium2 at RENDER_SCALE. Convert to grayscale → boost contrast 2× → sharpen. Run Tesseract. Filter words below confidence 30. Divide pixel coords by RENDER_SCALE to convert back to PDF-space units.
OCR step 1
Image preprocessing
Convert to grayscale → contrast × 2.0 via ImageEnhance → SHARPEN filter. Makes text edges crisper for better character recognition.
OCR step 2
Tesseract data extraction
image_to_data() returns per-word bounding boxes, text strings, and confidence scores. Words below conf = 30 are discarded.
OCR step 3
Coordinate rescaling
Divide pixel positions by RENDER_SCALE (1.5) to get PDF point coordinates. OCR words now share the same coordinate system as native words.
Unified word schema — both paths output:
text: "hello" x0: left edge top: top edge x1: right edge bottom: bottom edge page: 0-based index

5
Two-pass diff engine
Python LCS produces candidates — C++ Kv spatial engine refines them
Pass 1 — Python
LCS via difflib.SequenceMatcher
Compares flat word-text lists from old vs new commit. Returns opcodes: equal, replace, insert, delete.

• replace / delete → candidate removed words
• replace / insert → candidate added words

Fast but text-only — doesn't understand document structure.
Pass 2 — C++
Kv spatial engine via pybind11
Receives only LCS-flagged candidates. Groups words spatially into lines → sentences → paragraphs. Matches paragraphs by Jaccard similarity (threshold 0.55). For matched pairs: only changed words are highlighted. For unmatched: full block flagged.

Much more precise — avoids false positives from common words that shift position.
Why both passes are needed
LCS alone over-highlights common words
The word "the" appears thousands of times. LCS treats each occurrence as independent and flags position shifts as removals/additions. The Kv pass understands that "the" in paragraph 3 and "the" in paragraph 3 of the new file are the same word in the same place, and suppresses the false highlight. Result: only genuinely changed words glow.

Inside the C++ Kv engine — kv_mechanism.cpp
Three-phase spatial analysis compiled to a native Python extension via pybind11
①
Build Sentences
build_sentences()
Sort words by page → y-band (3pt) → x. Words within 3pt vertical tolerance merge into a Sentence with combined text, min/max bounds, x-start.
→
②
Build Paragraphs
build_paragraphs()
Adjacent sentences on same page within 20pt x-drift and 18pt y-gap merge into Paragraph. Sentences that indent or jump far → new paragraph.
→
③
Match Paragraphs
match_paragraphs()
Jaccard similarity between each old/new paragraph pair. ≥ 0.55 → matched pair, diff only changed words. < 0.55 → full block as add/remove.
For each old paragraph in the Kv engine:
Jaccard ≥ 0.55 → Matched pair
Build word sets for old and new paragraph. Words in old but not in new → added to removed list. Words in new but not in old → added to added list. Unchanged words: no highlight. Surgical precision.
Jaccard < 0.55 → Unmatched
Old paragraph has no match → every word → removed (yellow). New paragraph has no match → every word → added (blue). Treated as fully deleted or fully inserted block.
pybind11 bridge
Python ↔ C++ interface
PYBIND11_MODULE(kv_mechanism, m) registers run_kv_diff() as kv_mechanism.run_diff(). Compiles to a .so / .pyd shared library. Python passes lists of dicts; pybind11 handles all type conversion automatically.
Page normalization
normalize_pages() — why it's needed
LCS-flagged words from pages 40–45 are sent to C++. The engine's spatial grouping works best with 0-based pages. normalize_pages() subtracts the minimum page number before the call, then the offsets are added back to restore original page numbers for rendering.

Coordinate formula
Highlight rectangle math
x0_px = word.x0 × 1.5 − 2
y0_px = word.top × 1.5 − 2
x1_px = word.x1 × 1.5 + 2
y1_px = word.bottom × 1.5 + 2

The ±2px expansion ensures the highlight fully covers the glyph with a small visual margin.
Threading safety
Why the queue is essential
tkinter is not thread-safe. Only the main thread may draw to canvases. The worker thread puts rendered PIL Images into self.q (a thread-safe Queue). The main thread polls every 50ms via process_queue() → converts to PhotoImage → draws on canvas.

Queue message protocol — process_queue()
Seven message types the background worker sends to the UI thread
"status"
Updates status label in grey. Used for normal progress text like "Processing pages 1–10…"
"warn"
Updates status label in amber. OCR fallback notifications, page extraction errors, missing pytesseract.
"progress"
Sets progress bar value 0–100. Calculated as 5 + (chunk/total) × 85 per chunk.
"page_old"
Rendered PIL Image for left canvas. Converted to PhotoImage, appended to old_photos, drawn at old_y position.
"page_new"
Same for right canvas. new_y advances by image height + PAGE_GAP (28px) after each page is placed.
"done"
Shows "✅ Done — N removed · M added". Re-enables Compare button. Progress bar resets after 2s.
"error"
Shows red error message in status bar. Re-enables Compare. Full traceback printed to console via traceback.format_exc().
9
Complete data flow — end to end
Every transformation a single word goes through, from raw PDF to visible highlight
Step 1
PDF files on disk
Two files selected. Opened by both pdfplumber (text) and pypdfium2 (rendering). Both document handles stay open for the full worker thread lifetime, closed in the finally block.
Step 2
Per-page extraction → word dicts
pdfplumber native if ≥ 20 words. OCR via pypdfium2 + Tesseract if < 20. Both paths produce identical {text, x0, top, x1, bottom, page} dicts. The rest of the pipeline is path-agnostic.
Step 3
Tail merge + seam cut → committed word lists
Previous tail + current chunk = full working set. Seam algorithm finds the safe cut point. Words before seam = old_commit + new_commit — sent to the diff engine. Words after seam = new tail for the next iteration.
Step 4
LCS diff → candidate changed words
difflib.SequenceMatcher compares word-text sequences. Replace/delete opcodes → candidate removed words. Replace/insert opcodes → candidate added words. These lists are the inputs to the C++ pass.
Step 5
Kv spatial pass → final changed words
C++ engine builds sentences → paragraphs → Jaccard-matches pairs → returns only truly changed words with their original bounding box coordinates. Page numbers restored via + old_min_page / + new_min_page.
Step 6
Render pages → draw highlight overlays
pypdfium2 renders each page at 1.5×. PIL draws semi-transparent colored rectangles over each changed word's bounding box. Alpha composite merges page image + overlay into a final RGBA image.
Step 7
Queue → tkinter canvas → user sees result
Worker puts images in self.q. Main thread polls every 50ms, converts to ImageTk.PhotoImage, draws to canvas_old / canvas_new. Scroll region updated. Both panels scroll in sync via sync_offset_y.
